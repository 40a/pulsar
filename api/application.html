Applications API¶

This module implements the main classes for pulsar application framework. The framework is built on top of pulsar asynchronous engine and allows to implement servers with very little effort. The main classes here are: Application and MultiApp which, has the name suggests, is a factory of several Application running on a single server. The Configurator is a mixin used as base class for both Application and MultiApp.

Note

An instance of an Application is picklable and therefore can be sent from actor to actor using the actor message passing api.

Application¶

class pulsar.apps.Application(callable=None, load_config=True, **params)[source]¶

An application interface.

Applications can be of any sorts or forms and the library is shipped with several battery included examples in the pulsar.apps framework module.

These are the most important facts about a pulsar Application:

  • It derives from Configurator so that it has all the functionalities to parse command line arguments and setup the Configurator.cfg.
  • Instances must be picklable. If non-picklable data needs to be added on an Application instance, it should be stored on the local attribute dictionary (Application derives from pulsar.utils.log.LocalMixin).
  • Instances of an Application are callable objects accepting the calling actor as only argument.
  • When an Application is called for the first time, a new pulsar.Monitor instance is added to the pulsar.Arbiter, ready to perform its duties.
Parameters:
  • callable – Initialise the Application.callable attribute.
  • load_config – If False the Configurator.load_config() is not invoked. Default True.
  • params – Passed to the Configurator initialiser.
callable¶

Optional callable serving or configuring your application. If provided, the callable must be picklable, therefore it is either a function or a picklable object.

Default None

Multi App¶

class pulsar.apps.MultiApp(name=None, description=None, epilog=None, version=None, argv=None, parse_console=True, script=None, cfg=None, load_config=True, **kwargs)[source]¶

A MultiApp is a tool for creating several Application and starting them at once. It makes sure all settings for the applications created are available in the command line. The build() is the only method which must be implemented by subclasses. Check the examples.taskqueue.manage.server class in the taskqueue example for an example. The MultiApp derives from Configurator and therefore supports all its configuration utilities.

A minimal example usage:

import pulsar

class Server(pulsar.MultiApp):
    def build(self):
        yield self.new_app(TaskQueue)
        yield self.new_app(WSGIserver, prefix="rpc", callable=..., ...)
        yield self.new_app(WSGIserver, prefix="web", callable=..., ...)
build()[source]¶

Virtual method, must be implemented by subclasses and return an iterable over results obtained from calls to the new_app() method.

  • callable – optional callable (function of object) used during initialisation of App (the Application.callable).
  • params – additional key-valued parameters used when creating an instance of App.

Backend¶

class pulsar.apps.Backend(scheme, connection_string, name=None, **params)[source]¶

Base class for backends.

It is the base class for publish/subscribe backend and for the taskqueue backend.

A Backend is never initialised directly, the Backend.make() classmethod is used instead.

scheme¶

The scheme for this backend (local, redis, http, ...).

name¶

Optional name of the Application which created this Backend. If not available the arbiter is used as owner of this Backend.

params¶

Dictionary of additional parameters passed during initialisation. These parameters are first pre-processed by the setup() method.

connection_string¶

The connection string fro this backend. It is always in the form:

scheme:://address?param1=..&params2=... &...

where address is usually a host:port string.

A special connection string is:

local://&param1=...&...

which represents a backend implemented using pulsar itself. A local backend is installed in the monitor registered with name.

default_path¶

A class attribute which represents the path from where to load backend implementations.

id[source]¶

Identy for this backend.

It is calculated from the class name and the connection_string.

Therefore, if two backend instances from the same class have the same connection string, than the id is the same.

classmethod make(backend=None, name=None, **kwargs)[source]¶

Create a new Backend from a backend connection string which is of the form:

scheme:://host?params

For example, a parameter-less local backend is:

local://

A redis backend could be:

redis://127.0.0.1:6379?db=1&password=bla
Parameters:
  • backend – the connection string, if not supplied local:// is used.
  • kwargs – additional key-valued parameters used by the Backend during initialisation.

Create the connection string used during initialisation of a new Backend.
