Settings¶

This is the full list of pulsar settings available for configuring your pulsar server or applications. Pulsar can run several pulsar.apps.Application in one server, via the pulsar.apps.MultiApp class. Each application has its own set of configuration parameters with the only exception of the global server settings.

Global Server Settings¶

These settings are shared by the arbiter as well as all pulsar workers. They set global settings the debug mode, the proxy server to use when making HTTP requests, and server specific configuration parameters.

config¶

Command line: -c, --config

Default: config.py

The path to a Pulsar config file, where default Settings paramaters can be specified.

http_proxy¶

Command line: --http-proxy

Default: ````

The HTTP proxy server to use with HttpClient.

http_py_parser¶

Command line: --http-py-parser

Default: False

Set the python parser as default HTTP parser.

debug¶

Command line: --debug

Default: False

Turn on debugging in the server.

This limits the number of worker processes to 1 and changes some error handling that’s sent to clients.

daemon¶

Command line: -D, --daemon

Default: False

Daemonize the Pulsar process (posix only).

Detaches the server from the controlling terminal and enters the background.

pidfile¶

Command line: -p, --pid

Default: None

A filename to use for the PID file.

If not set, no PID file will be written.

password¶

Command line: --password

Default: None

Set a password for the server

user¶

Command line: -u, --user

Default: None

Switch worker processes to run as this user.

A valid user id (as an integer) or the name of a user that can be retrieved with a call to pwd.getpwnam(value) or None to not change the worker process user.

group¶

Command line: -g, --group

Default: None

Switch worker process to run as this group.

A valid group id (as an integer) or the name of a user that can be retrieved with a call to pwd.getgrnam(value) or None to not change the worker processes group.

loglevel¶

Command line: --log-level

Default: info

The granularity of log outputs.

Valid level names are:

  • debug
  • info
  • warning
  • error
  • critical

loghandlers¶

Command line: --log-handlers

Default: ['console']

log handlers for pulsar server

logconfig¶

Default: {}

The logging configuration dictionary.

This settings can only be specified on a config file

process_name¶

Command line: -n, --name

Default: None

A base to use with setproctitle for process naming.

This affects things like ps and top. If you’re going to be running more than one instance of Pulsar you’ll probably want to set a name to tell them apart. This requires that you install the setproctitle module.

It defaults to ‘pulsar’.

default_process_name¶

Default: Pulsar

Internal setting that is adjusted for each type of application.

Worker Processes¶

This group of configuration parameters control the number of actors performing for a given pulsar.Monitor, the type of concurreny of the server and other actor-specific parameters.

workers¶

Command line: -w, --workers

Default: 1

The number of workers for handling requests.

If using a multi-process concurrency, a number in the the 2-4 x NUM_CORES range should be good. If you are using threads this number can be higher.

concurrency¶

Command line: --concurrency

Default: process

The type of concurrency to use: process or thread.

max_requests¶

Command line: --max-requests

Default: 0

The maximum number of requests a worker will process before restarting.

Any value greater than zero will limit the number of requests a worker will process before automatically restarting. This is a simple method to help limit the damage of memory leaks.

If this is set to zero (the default) then the automatic worker restarts are disabled.

backlog¶

Command line: --backlog

Default: 2048

The maximum number of concurrent requests.

This refers to the number of clients that can be waiting to be served. Exceeding this number results in the client getting an error when attempting to connect. It should only affect servers under significant load. Must be a positive integer. Generally set in the 64-2048 range for socket servers, 5-10 for task-queue servers.

timeout¶

Command line: -t, --timeout

Default: 30

Workers silent for more than this many seconds are killed and restarted.

Application Hooks¶

Application hooks are functions which can be specified in a config file to perform custom tasks in a pulsar server. These tasks can be scheduled when events occurs or at every event loop of the various components of a pulsar application.

All application hooks are functions which accept one parameter only, the actor invoking the function.

post_fork¶

Default: pass_through()

Called just after a worker has been forked.

The event loop is not yet available.

when_ready¶

Default: pass_through()

Called just before a worker starts its event loop.

This is a chance to setup pulsar.EventLoop callbacks which can run periodically, at every loop or when some defined events occur.

connection_made¶

Default: pass_through()

Called after a new connection is made.

The callable needs to accept one instance variables for the connection instance.

connection_lost¶

Default: pass_through()

Called after a connection is lost.

The callable needs to accept one instance variables for the connection instance.

pre_request¶

Default: pass_through()

Called just before a worker processes the request.

The callable needs to accept two instance variables for the Worker and the Request.

post_request¶

Default: pass_through()

Called after a worker processes the request.

The callable needs to accept two instance variables for the Worker and the Request.

Socket Servers¶

bind¶

Application namespace: socket

Command line: -b, --bind

Default: 127.0.0.1:8060

The socket to bind.

A string of the form: HOST, HOST:PORT, unix:PATH. An IP is a valid HOST.

keep_alive¶

Application namespace: socket

Command line: --keep-alive

Default: 15

The number of seconds to keep an idle client connection open.

Task Consumer¶

This section covers configuration parameters used by CPU bound type applications such as the distributed task queue and the test suite.

task_backend¶

Application namespace: tasks

Command line: --task-backend

Default: local://

Task backend.

A task backend is string which connect to the backend storing Tasks) which accepts one parameter only and returns an instance of a distributed queue which has the same API as pulsar.MessageQueue. The only parameter passed to the task queue factory is a pulsar.utils.config.Config instance. This parameters is used by pulsar.apps.tasks.TaskQueue application.

task_paths¶

Application namespace: tasks

Default: []

List of python dotted paths where tasks are located.

This parameter can only be specified during initialization or in a config file.

schedule_periodic¶

Application namespace: tasks

Command line: --schedule-periodic

Default: False

Enable scheduling of periodic tasks.

If enabled, pulsar.apps.tasks.PeriodicJob will produce tasks according to their schedule.

Test¶

verbosity¶

Application namespace: test

Command line: --verbosity

Default: 1

Test verbosity, 0, 1, 2, 3

test_timeout¶

Application namespace: test

Command line: --test-timeout

Default: 30

Tests which take longer than this many seconds are timed-out and failed.

labels¶

Application namespace: test

Default: None

Optional test labels to run. If not provided all tests are run.

To see available labels use the -l option.

size¶

Application namespace: test

Command line: --size

Default: normal

Optional test size.

list_labels¶

Application namespace: test

Command line: -l, --list-labels

Default: False

List all test labels without performing tests.

sequential¶

Application namespace: test

Command line: --sequential

Default: False

Run test functions sequentially. Don’t run them asynchronously.

Utilities¶

pulsar.utils.config.pass_through(arg)¶

A dummy function accepting on parameter only. It does nothing. It is used as default by Application Hooks.
